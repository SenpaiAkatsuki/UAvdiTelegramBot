# End-to-End Test Plan

## Scope

This checklist validates the full lifecycle:

- application intake (bot + site),
- moderation and binding,
- LiqPay payment and callback processing,
- subscription and group access gating.

Smoke tests remain optional and are enabled only when `RUN_WEBLIUM_SMOKE_TESTS=1`.

## Preconditions

1. Start PostgreSQL (Docker Compose or local).
2. Start webhook app:
   - `python -m infrastructure.api.weblium_app`
3. Start bot polling:
   - `python bot.py`
4. Configure `.env`:
   - `WEBHOOK_ENABLED=True`
   - valid `WEBHOOK_WEBLIUM_SECRET`
   - valid LiqPay sandbox keys and `PUBLIC_WEBHOOK_BASE_URL`
   - correct `CHAT_MEMBERSHIP_CHAT_ID` and `VOTING_CHAT_ID`

## Cases

1. Bot-originated application with `tg_token`
- Steps:
  - User starts bot and clicks application link generated by bot.
  - Submit website form including `tg_token` in URL.
- Expected:
  - Webhook creates matched application.
  - User gets "under review" Telegram message.
  - Group vote is created for matched branch.

2. Site-originated application without `tg_token`
- Steps:
  - Submit website form directly (no bot link/token).
- Expected:
  - Webhook creates unlinked application.
  - Bind token is generated and included in admin vote payload.

3. Matched webhook flow
- Steps:
  - Complete vote to approved for matched application.
- Expected:
  - Application status moves to payment-eligible state.
  - User can open pay flow.

4. Unlinked webhook flow
- Steps:
  - Submit unlinked form, approve content in admin flow.
- Expected:
  - Application remains unbound until user self-binds.
  - Payment is blocked before bind completion.

5. Admin content approval for unlinked applications
- Steps:
  - In admin flow, approve unlinked submission content.
- Expected:
  - Content moderation state is approved.
  - Application is available for later bind by a real user.

6. Self-bind by phone/email
- Steps:
  - User runs bind flow and provides matching phone/email.
- Expected:
  - Unlinked application is attached to correct `tg_user_id`.
  - Bind token/user mapping cannot be reused after successful bind.

7. Payment allowed only after proper bind + approval
- Steps:
  - Try payment before approval/bind.
  - Try payment after approval + bind.
- Expected:
  - Before: payment creation rejected.
  - After: payment row is created and pay link is returned.

8. Payment idempotency
- Steps:
  - Complete one LiqPay payment.
  - Re-deliver same successful callback payload.
- Expected:
  - Payment stays `PAID`.
  - Subscription is extended only once.
  - Callback audit is updated without duplicate extension.

9. Join request gating
- Steps:
  - Try `Get group access` with inactive/expired subscription.
  - Try with active subscription.
- Expected:
  - Inactive/expired: blocked with renew message.
  - Active: invite link is issued, user can join group.

10. Duplicate webhook delivery (Weblium)
- Steps:
  - Send identical webhook payload twice.
- Expected:
  - First request ingested.
  - Second request returns duplicate-ignored result.
  - No duplicate application records.

11. Reused token
- Steps:
  - Submit one form with valid `tg_token`.
  - Reuse same token in another webhook submit.
- Expected:
  - First one is processed.
  - Reused token is treated as invalid/used and does not create second matched bind.

12. Invalid token
- Steps:
  - Submit webhook with random/expired `tg_token`.
- Expected:
  - Request is accepted as unlinked flow (not matched).
  - Bind token is generated for manual/user bind.

13. Leaked invite link
- Steps:
  - Share invite link with another account and try to reuse.
- Expected:
  - One-time link should not grant unlimited access.
  - Final access still controlled by membership handlers and chat permissions.

## Optional smoke tests

Run only when explicitly enabled:

- PowerShell:
  - `$env:RUN_WEBLIUM_SMOKE_TESTS=1; pytest tests/smoke -q`
- Bash:
  - `RUN_WEBLIUM_SMOKE_TESTS=1 pytest tests/smoke -q`

When flag is not set, smoke tests are skipped by design.
